# Self-Host Deployment Contract

> 🇪🇸 Versión en español: [../es/deployment-self-host.md](../es/deployment-self-host.md)

This is the **platform-agnostic deployment contract** for Vaultly. It tells you exactly what the app needs to run, regardless of where you run it (Kubernetes, Nomad, Docker Swarm, ECS, `docker run` on a VPS, etc.).

**This doc does NOT include**:
- Kubernetes manifests, Helm charts, or Kustomize bases (write your own from the contract below)
- ArgoCD / Flux / Jenkins X / Spinnaker / Tekton-specific instructions
- Cloud-provider-specific glue (use this together with [connecting-cloud-databases.md](connecting-cloud-databases.md))

If you want a managed PaaS template instead, use [deployment-railway.md](deployment-railway.md).

---

## 1. The two images

Vaultly ships as two Docker images:

| Image | Built from | Purpose | Listens on |
|-------|-----------|---------|------------|
| API | [`apps/api/Dockerfile`](../../apps/api/Dockerfile) | NestJS backend, runs migrations, serves REST + SSE | `3000` (HTTP) |
| Web | [`apps/web/Dockerfile`](../../apps/web/Dockerfile) | nginx serving the built React SPA | `80` (HTTP) |

Build them yourself from the repo, or pull from your registry once your CI publishes them.

```bash
# Local build for testing
docker build -t vaultly-api:local -f apps/api/Dockerfile .
docker build -t vaultly-web:local -f apps/web/Dockerfile .
```

Both Dockerfiles expect the **monorepo root** as the build context (they read `pnpm-workspace.yaml` and `pnpm-lock.yaml`).

## 2. Environment variables — the contract

Full reference: [environment-variables.md](environment-variables.md). The contract below is the minimum to start.

### API (runtime)

| Variable | Required | Notes |
|----------|----------|-------|
| `DATABASE_URL` | yes | Control DB connection string (must be PostgreSQL 16+) |
| `NODE_ENV` | yes | `production` in any deployed environment |
| `PORT` | no (default `3000`) | Listen port |
| `CORS_ORIGIN` | yes in production | Exact frontend domain, no wildcards |
| `KEYCLOAK_URL` | yes | External Keycloak instance |
| `KEYCLOAK_REALM` | yes | Realm name, case-sensitive |
| `KEYCLOAK_CLIENT_ID` | yes | API client ID in Keycloak |
| `R2_ACCOUNT_ID` | yes in production | Cloudflare account ID (32-char hex) |
| `R2_ACCESS_KEY_ID` | yes in production | From R2 API Token |
| `R2_SECRET_ACCESS_KEY` | yes in production | From R2 API Token |
| `R2_BUCKET_NAME` | yes in production | Bucket for dumps |

### Web (build-time AND runtime)

The Web layer is **GitOps-friendly by design**: it reads runtime config from `window.APP_CONFIG`, which is generated by `entrypoint.sh` at container startup from environment variables. **No `.env` file goes into the image.**

| Variable | Required | Notes |
|----------|----------|-------|
| `VITE_API_URL` | yes | Backend URL (e.g., `https://api.vaultly.example.com`) |
| `VITE_KEYCLOAK_URL` | yes | Same as API's `KEYCLOAK_URL` |
| `VITE_KEYCLOAK_REALM` | yes | Same as API's `KEYCLOAK_REALM` |
| `VITE_KEYCLOAK_CLIENT_ID` | yes | Web client ID in Keycloak (different from API's) |
| `VITE_APP_BASE_URL` | yes | Public URL of the web (e.g., `https://vaultly.example.com`) |

> The `VITE_*` prefix is historical (these were Vite build-time vars). At runtime they are read from the same names as container env vars by `entrypoint.sh`. You do not need to rebuild the image to change them — just restart the container with new env values.

## 3. Health endpoints and probes

| Endpoint | Path | Auth | Returns |
|----------|------|------|---------|
| API liveness/readiness | `GET /health` | None (public) | `200` if healthy |
| Web liveness | `GET /` | None | `200` (nginx serves index.html) |

### Recommended probe configuration

| Probe | API | Web |
|-------|-----|-----|
| Liveness path | `/health` | `/` |
| Readiness path | `/health` | `/` |
| Initial delay | 15s (allow migrations to run) | 5s |
| Period | 10s | 10s |
| Timeout | 3s | 2s |
| Failure threshold | 3 | 3 |

The **15-second initial delay on the API** is important: on first boot, TypeORM runs migrations against the control DB before the HTTP server is fully ready. If your probe runs too soon, the pod gets killed before migrations finish.

## 4. Resource sizing

Honest baseline numbers. **Validate against your own workload** — they are starting points, not commitments.

| Container | CPU request | CPU limit | Memory request | Memory limit |
|-----------|------------|-----------|----------------|--------------|
| API | 100m | 500m | 256 MiB | 512 MiB |
| Web (nginx) | 25m | 100m | 32 MiB | 128 MiB |
| Control DB (Postgres 16) | 250m | 1000m | 512 MiB | 2 GiB |

**Scaling drivers**:
- API memory grows with **dump size during streaming**. `pg_dump` of a 50GB DB does not load it all into memory (streaming), but parts of the multipart R2 upload do buffer. If you back up large DBs (>20GB), bump API memory to 1 GiB.
- API CPU spikes during dump compression and restore. Brief spikes are fine; sustained pressure means you're undersized.
- Web is essentially free — nginx serving static files. If you have thousands of concurrent users, that's another conversation, but for an internal DevOps tool, the defaults are generous.

## 5. Storage

| Container | Persistent volume needed? | Why |
|-----------|---------------------------|-----|
| API | **No** | Stateless. Dumps stream to R2, nothing on local disk. |
| Web | **No** | Stateless. Just serves static files. |
| Control DB | **Yes** | This is the source of truth — back it up too (see [devops-runbook.md §5](devops-runbook.md#5-backup-of-the-control-database-itself)). |

For the control DB, use the volume strategy of your platform: PVC on K8s, EBS on EC2, local SSD with daily snapshots on VPS, etc. **Minimum 20 GiB** for the control DB volume; grows slowly (audit logs are the main driver).

## 6. Networking

### Ingress

| Service | Ingress needed | Protocol | Notes |
|---------|----------------|----------|-------|
| Web | yes | HTTPS (terminate at ingress) | This is the user-facing entry |
| API | yes | HTTPS | The Web hits it; if Web and API are on the same domain, route `/api/*` to API |
| Control DB | no | — | Internal only, never expose publicly |

Recommended: put both behind the same hostname (`vaultly.example.com` for Web, `vaultly.example.com/api` for API) to avoid CORS gymnastics. If you split (`app.` and `api.`), set `CORS_ORIGIN` on the API to the Web's exact domain.

### Egress

The API needs outbound network access to:

- **Keycloak** — wherever you host it (HTTPS, port 443)
- **Cloudflare R2** — `*.r2.cloudflarestorage.com` (HTTPS, port 443)
- **Every managed/on-prem DB** you register as a connection (TCP, varies — typically 5432/3306)

If your egress is firewalled, allowlist these explicitly. The Web container has **no outbound network needs** — it just serves static files.

## 7. Scaling constraints — READ THIS

> ⚠️ **The API is single-replica only, today.**

The cronjob scheduler runs in-process (`@nestjs/schedule`). If you deploy the API with `replicas: 2` (or more), **every replica fires every scheduled backup**, producing duplicate dumps and wasted R2 storage.

| Service | Max replicas | Min replicas | Why |
|---------|-------------|--------------|-----|
| API | **1** | 1 | In-process scheduler |
| Web | unlimited | 1 | Pure nginx, fully stateless |
| Control DB | 1 primary (+ replicas if you want HA reads, but not required) | 1 | Standard Postgres |

For HA on the API today: configure a single replica with `RollingUpdate` strategy (`maxSurge: 1, maxUnavailable: 0`). On K8s this means brief unavailability during deploys; if that's unacceptable, **the BullMQ + Redis migration in [architecture-roadmap.md §4](architecture-roadmap.md#phase-5--operational-hardening-independent-of-driver-work) is the unblocker**, not multi-replica today.

## 8. Migration behavior

The API runs migrations on startup with `migrationsRun: true`. Implications:

- **No init container needed**. The API container handles migrations itself.
- **First boot against an empty DB will fail** until [`InitialSchema` migration](database-migrations.md#3-the-solution-generate-an-initialschema-migration) is generated.
- **Rolling deploys with migrations**: TypeORM acquires an advisory lock on the `migrations` table, so concurrent startups are safe — but since you only run one replica, this is moot.
- **Rollbacks are non-trivial**. A migration that adds a column is forward-compatible (the old image ignores the new column). One that drops a column or changes a type is not. Plan migrations to be backward-compatible for at least one release.

## 9. The Keycloak gotcha

Railway has a pre-built Keycloak template that drops in with three clicks. **In self-host, you do NOT get that.** You need to:

1. Deploy Keycloak yourself (Docker image: `quay.io/keycloak/keycloak`).
2. Configure its own Postgres database.
3. Create the realm (e.g., `vaultly-control`).
4. Create both clients (`vaultly-api`, `vaultly-web`) with PKCE enabled.
5. Configure the redirect URIs to point at your Web domain.

Realm and client setup details are documented per-step in [deployment-railway.md §Configuring the realm and clients](deployment-railway.md#configuring-the-realm-and-clients-in-keycloak) — the steps are the same regardless of where Keycloak runs.

If you want a documented Keycloak realm export to import, that does not exist today. It's a reasonable feature to add — open an issue if you need it.

## 10. Minimum viable Kubernetes example (illustrative, NOT a template)

Below is the **smallest possible** set of manifests to run Vaultly on K8s. It is intentionally minimal and **does not include** ingress, TLS, secrets management, persistent volumes, or any production hardening. Treat it as a starting point to verify the contract, not as something to deploy as-is.

```yaml
# api-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vaultly-api
spec:
  replicas: 1  # MUST be 1 — see §7
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: vaultly-api
  template:
    metadata:
      labels:
        app: vaultly-api
    spec:
      containers:
        - name: api
          image: ghcr.io/your-org/vaultly-api:0.1.0
          ports:
            - containerPort: 3000
          envFrom:
            - secretRef:
                name: vaultly-api-secrets
          resources:
            requests:
              cpu: 100m
              memory: 256Mi
            limits:
              cpu: 500m
              memory: 512Mi
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 10
---
# api-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: vaultly-api
spec:
  selector:
    app: vaultly-api
  ports:
    - port: 3000
      targetPort: 3000
```

The Web is structurally identical — swap `3000` for `80`, point env vars at the API service, done. You handle ingress, TLS, secrets, and persistence with whatever your team already uses.
