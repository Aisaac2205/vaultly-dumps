# Conventions — Vaultly Control

> 🇪🇸 Versión en español: [../es/conventions.md](../es/conventions.md)

## File naming

- API modules: flat names inside their domain folder — `service.ts`, `controller.ts`, `module.ts`. Context comes from the folder (e.g. `modules/backup/service.ts`).
- React components: PascalCase — `StatusBadge.tsx`, `Layout.tsx`.
- Hooks: camelCase with `use` prefix — `useConnections.ts`, `useBackupHistory.ts`.
- Backend enums: `kebab-case.enum.ts` under `apps/api/src/database/enums/` (e.g. `backup-category.enum.ts`). These are the source of truth for cross-app values.
- Frontend types that mirror backend enums: `kebab-case.types.ts` under `apps/web/src/types/` or `features/<feature>/types.ts`.

---

## Imports

- Never import across apps directly (`apps/web` does not import from `apps/api`).
- There is no shared package. Cross-app contracts (enums / string unions used in DTOs, query params, payloads) are **duplicated**:
  - **Backend** is the source of truth: TypeScript enums in `apps/api/src/database/enums/`, validated at runtime with `@IsEnum` from class-validator.
  - **Frontend** replicates the literal values as union types in `apps/web/src/types/` or `features/<feature>/types.ts`.
- If you add a new value to a backend enum, mirror it in the frontend union type **in the same PR**. The `@IsEnum` validator will reject any request that ships an unsynced value.

---

## Environment variables

- Never commit `.env` files with real values — they are in `.gitignore`.
- The `.env.example` files are the source of truth for which variables exist.
- Frontend variables must start with `VITE_` so Vite injects them into the bundle.
- On Railway: configure them as Service Variables. `VITE_*` are passed automatically as build args when the Dockerfile declares them as `ARG`.

## Commits — Conventional Commits

### Types that trigger a release

| Type | Bump | When to use it |
|------|------|----------------|
| `fix:` | patch (0.1.0 → 0.1.1) | Bug fix |
| `perf:` | patch (0.1.0 → 0.1.1) | Performance improvement, no API change |
| `feat:` | minor (0.1.0 → 0.2.0) | New user-visible functionality |
| `feat!:` | major (0.1.0 → 1.0.0) | Breaking change |

**The test for `feat:`**: can the user do something they COULDN'T do before? Yes → `feat:`. No (wiring, extraction, migration, polish) → `refactor:` (no release).

### Types that do NOT trigger a release (free use)

`chore:`, `docs:`, `refactor:`, `build:`, `ci:`, `style:`, `test:` — useful for a clean history but they do not bump the version.

### Strict rule: NO parentheses or scopes

```
✅ fix: handle connection timeout gracefully
✅ feat: add on-demand backup endpoint
✅ feat!: change R2 path structure
✅ chore: upgrade NestJS to 11.0.7
✅ refactor: simplify auth guard session validation

❌ fix(connections): handle connection timeout gracefully
❌ feat(backup): add on-demand backup endpoint
❌ chore(deps): upgrade NestJS
```

**Why no scopes**: the subject already describes the *what*; the diff path describes the *where*. Parentheses add visual noise without information the reviewer cannot see better in `git show`.

### Breaking changes

Two valid forms:

1. **Bang**: `feat!: drop support for Node 20`
2. **Body trailer**:
   ```
   feat: redesign auth flow

   BREAKING CHANGE: tokens now expire after 1h instead of 24h.
   Clients must implement refresh logic.
   ```

For major releases always include the trailer with an explanation, even if you already used `!`. The changelog generated by semantic-release reads the trailer.

## TypeScript

Two-tier config:

**`tsconfig.base.json` (root)** — shared options:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "declaration": true,
    "sourceMap": true
  }
}
```

**Each app extends the base:**

- `apps/api` — adds `"module": "CommonJS"`, `emitDecoratorMetadata: true`, `experimentalDecorators: true` (required by NestJS).
- `apps/web` — adds `"module": "ESNext"`, `"moduleResolution": "bundler"`, `"jsx": "react-jsx"` (Vite bundler mode).
